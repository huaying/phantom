# Phantom encoder benchmarks — 2026-04-20

phantom-bench v0.4.10 on the two GPU test VMs. Component bench measures the
encoder in isolation (one synthetic BGRA frame per round with
per-frame randomization so P-frames can't cheat). Fused bench measures
the full `NVFBC → NVENC` zero-copy path with a live X desktop and
`xdotool` jiggle forcing screen updates.

All numbers are the mean of 30 rounds after 3 warmup rounds, plus p50 /
p95. `phantom-server` was paused during the NVFBC section because NVFBC
only allows one client at a time.

Both VMs ran the same binary (built on U22, glibc 2.35) to isolate GPU /
driver differences from build-time variance.

## Takeaways

**Fused zero-copy path is 3×+ faster than standalone NVENC.**  
L40: 2.83 ms fused vs 9.1 ms standalone at 1080p 5 Mbps → **3.2× speedup**. The
`NVFBC grab → CUdeviceptr → NVENC` hop avoids the BGRA→NV12 SIMD +
`cuMemcpyHtoD` that the standalone path does, which together account for
most of the ~6 ms gap.

**L40 is ~30 % faster than A40 on fused H.264.**  
2.83 ms vs 4.16 ms at 1080p 5 Mbps. Ada's updated NVENC unit shows the
expected generational gain.

**OpenH264 CPU blows past the 33 ms frame budget at 1440p.**  
75+ ms at 1440p on both machines regardless of bitrate — single-threaded
software encode can't keep up with 30 fps at that resolution. For 1440p
+ 30 fps you need NVENC (or any hardware encoder), no way around it.

**AV1 encode time on L40 is ~same as H.264**, sometimes slightly
faster (2.40 ms vs 2.64 ms fused). AV1's compression win shows up in
bytes, not CPU/GPU time — the fused numbers look similar because the
noise-pattern test frame isn't representative of real screen content
(lots of entropy, no reuse). Bytes column is more interesting: **21 K
AV1 vs 92 K H.264 at 1080p 5 Mbps** in the component bench — AV1
reaches the rate-control target 4× faster, which is the bandwidth story
we want to ship when client decode support catches up (task #34).

## Methodology notes + caveats

- **Component bench uses a synthetic per-frame-noise pattern.** That
  pattern has much more entropy than a typical desktop, so byte counts
  are higher than you'd see streaming a real session. Absolute numbers
  don't transfer; ratios between encoders / resolutions do.
- **Fused bench uses the actual X desktop** plus `xdotool` jiggling the
  cursor to force NVFBC to return frames. NVFBC's default NOWAIT mode
  returns `None` on a static screen; the jiggle is a workaround (see
  `docs/pitfalls.md`). Bench wraps it in a `Drop` guard so Ctrl+C /
  panic still kill xdotool — otherwise stray `xdotool mousemove` loops
  would drift any real phantom session on the same display.
- **phantom-server must be paused** during the NVFBC section (NVFBC
  only allows one client). We SIGSTOP it before the bench and SIGCONT
  after.
- **NVENC "avg bytes/frame" looks suspiciously high** (92 K at 1080p 3
  Mbps when the math says ~12 K/frame). Reason: first frame is always a
  keyframe (SPS/PPS + IDR ≈ 20-40 K), and the bench runs 30 rounds so
  the keyframe isn't amortized as heavily as a long streaming session.
  Plus NVENC's rate control takes a few frames to tune in, and the
  noise pattern gives it nothing to reuse.
- **Encode times include Rust FFI overhead** to the NVENC C API.
  Typically <100 μs, not significant relative to the ~3 ms encode
  itself.

## Reproducing

```bash
# On any Linux + NVIDIA box:
git clone https://github.com/huaying/phantom.git
cd phantom
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
sudo apt-get install -y libxcb1-dev libxcb-shm0-dev libxcb-randr0-dev \
    libxdo-dev libpulse-dev libopus-dev libasound2-dev xdotool nasm cmake
cargo build --release -p phantom-bench
# Pause phantom-server if running (it holds NVFBC):
#   sudo pkill -STOP -x phantom-server
DISPLAY=:0 XAUTHORITY=/run/user/$UID/gdm/Xauthority ./target/release/phantom-bench
#   sudo pkill -CONT -x phantom-server
```

DXGI→NVENC fused bench (Windows) exists in the same binary but wasn't
run for this report. Windows test VMs (Win10/Win11) exist — follow-up.
